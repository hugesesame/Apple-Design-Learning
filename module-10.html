Module 10 · アクセシビリティ

アクセシビリティ

Appleはアクセシビリティを「義務」ではなくデザインの完成度として扱う。すべての人が使えることが、すべての人にとって良いデザインだ。

なぜアクセシビリティが「品質」なのか

アクセシビリティを「障害者向けの追加対応」と捉えるのは誤りだ。日本では高齢者人口が28%を超えており、視力低下・手の震え・認知的疲労を持つユーザーは全人口の多数を占める。

Appleがアクセシビリティに投資し続ける理由は明確だ。アクセシブルなデザインは、すべてのユーザーにとって使いやすいデザインと一致するからだ。コントラスト比が高いテキストは、晴天下でiPhoneを使うユーザーにも読みやすい。

アクセシビリティは
デザインの制約ではなく、
デザインの完成を確認する基準だ。

Apple Accessibility — Design for Everyone

WCAGとAppleの関係

WCAG（Web Content Accessibility Guidelines）はアクセシビリティの国際基準だ。AppleのウェブプロパティはWCAG 2.1 AAを目標として設計されている。

レベルコントラスト比（通常テキスト）コントラスト比（大テキスト）Appleの対応
A なし（規定なし） なし 最低限
AA 4.5 : 1 以上 3 : 1 以上 目標基準
AAA 7 : 1 以上 4.5 : 1 以上 主要テキストは多くがこれを達成

大テキストの定義：18pt（24px）以上、または14pt（18.67px）以上のボールド体。design.mdのtype scaleではdisplay-sm（21px）以上が「大テキスト」に該当する。

ink-muted-48 の扱い

#7a7a7a は白背景で約4.48:1のコントラスト比だ。WCAG AA（4.5:1）をわずかに下回る。そのためdesign.mdは ink-muted-48 を補助情報・キャプション・ラベル（=大テキスト扱いが多い文脈）に限定して使用する。本文テキストには使わない。

フォーカス管理

キーボードのみでサイトを操作するユーザーにとって、フォーカスリングは「今どこにいるか」を示す唯一の手がかりだ。

誤り — outline:none

キーボードユーザーが迷子になる

正しい — outline visible

3px solid #0071e3 · offset 3px

/* フォーカスリング — Appleの実装 */ :focus-visible { outline: 3px solid var(--color-primary-focus); outline-offset: 3px; border-radius: 4px; } /* :focus ではなく :focus-visible を使う */ /* マウスクリック時はフォーカスリングを非表示にし */ /* キーボード操作時のみ表示する */

:focus-visible はモダンブラウザでサポートされており、:focus と異なり「マウスクリック後のリング表示」を防ぎながら「キーボード操作時のリング表示」を維持する。

セマンティックHTML

スクリーンリーダーはHTMLの意味を読み上げる。<div> のクリックハンドラはスクリーンリーダーに「ボタン」として認識されない。

誤り
<div onclick="buy()">
  購入する
</div>
正しい
<button type="button"
  class="button-primary"
  onclick="buy()">
  購入する
</button>
要素用途スクリーンリーダー読み上げ
<nav>ナビゲーション領域「ナビゲーション、ランドマーク」
<main>ページのメインコンテンツ「メイン、ランドマーク」
<button>アクションを起こすUI「購入する、ボタン」
<a href>他URLへの遷移「詳しく見る、リンク」
<h1〜h6>見出し階層「iPhoneがやってきた、見出しレベル1」
<img alt>画像コンテンツalt属性のテキストを読み上げ

alt テキスト

画像のalt属性はスクリーンリーダーユーザーへの「画像の説明」だ。ただし、すべての画像にテキストを書けばよいわけではない。装飾画像は alt="" にする。

シーン悪い例良い例
製品写真 "image001.jpg" "iPhone 16 Pro — チタニウムブラック、斜め正面ビュー"
アイコン付きボタン "アイコン" aria-label="カートに追加"（imgはalt=""）
背景装飾 "blue gradient background" alt=""（装飾はalt空）
ロゴ "logo" "Apple"（社名そのもの）
グラフ・図解 "グラフ" "パフォーマンス比較グラフ：M4は前世代比40%高速"

ARIAとスクリーンリーダー

ARIAはHTMLの意味を補完するための属性セットだ。セマンティックHTMLで解決できる場合はARIAを使う必要はない。ARIAはセマンティックHTMLの代替ではなく補完だ。

aria-label ボタンのテキストが視覚的アイコンのみの場合に読み上げラベルを提供。例: aria-label="カートに追加"
aria-hidden スクリーンリーダーから要素を隠す。装飾アイコン・繰り返しコンテンツに使用。例: aria-hidden="true"
aria-expanded 開閉可能なパネルの状態を示す。例: ハンバーガーメニューが開いているとき aria-expanded="true"
aria-live 動的に更新されるコンテンツをスクリーンリーダーに通知。例: フォームのエラーメッセージ領域に aria-live="polite"

スクリーンリーダーでテストする

Macに標準搭載されているVoiceOver（⌘＋F5 または Fn＋⌘＋F5）でページを操作してみることが最良のテストだ。Tab キーでフォーカスを移動し、すべての操作が音声だけで完結できるか確認する。

演習

アクセシビリティ監査の実施

あなたの手元のHTMLページまたはapple.comを対象に、アクセシビリティ監査を実施してください。

  1. ChromeのDevTools「Lighthouse」タブを開き、アクセシビリティスコアを計測する。スコアと問題の一覧をメモする。
  2. Tabキーのみでページ全体を操作する。フォーカスリングが消えるか、操作できない要素があるかを記録する。
  3. MacのVoiceOver（⌘＋F5）を起動し、主要なCTAボタンを音声で操作する。適切なラベルが読み上げられるか確認する。
  4. WebAIM Contrast Checkerで自分のデザインの主要3色ペアのコントラスト比を計測し、AAをすべて通過させる。
  5. 発見した問題を修正し、修正前後のLighthouseスコアをコミュニティに投稿する。
