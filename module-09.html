Module 09 · モーション

モーションとアニメーション

Appleのアニメーションは「装飾」ではない。情報の遷移を助け、ユーザーの注意を適切に誘導する機能的なツールだ。

このモジュールで学ぶこと

apple.comは控えめだがきわめて精密なアニメーションで動く。製品ページのスクロールで製品が「浮かび上がる」感覚、ボタンを押したときの「押し込まれる」確かな応答感——これらはすべて設計されたモーションだ。

このモジュールでは、Appleが採用するイージング関数・デュレーション・アクティブ状態のアニメーション、そしてアクセシビリティへの配慮（Reduced Motion）を学ぶ。

大前提

アニメーションは「気づかれないこと」が最高の状態だ。ユーザーが「このアニメーションすごい」と感じた時点で、アニメーションはコンテンツの邪魔をしている。

モーション哲学

Appleのモーションには3原則がある。

1
自然（Natural）
物理法則に従うモーション。ease-out（始めが速く、終わりが遅い）は現実世界の摩擦をシミュレートする。
2
機能的（Functional）
モーションは目的を持つ。「どこから来たか」「どこへ行くか」を伝える。装飾のためのアニメーションは存在しない。
3
控えめ（Restrained）
短く、速く、静かに。デュレーションは最大でも0.6秒。注意を奪わない。

アニメーションは「製品を見せる」のではなく
「製品を理解させる」ために存在する。

Apple Human Interface Guidelines — Motion

イージング関数

Appleのモーションで最も多用されるイージングは ease-out だ。スプリングアニメーション（iOS的）はweb上ではcubic-bezierで近似する。

Linear（禁止）
機械的。Appleでは使わない。
ease-in（禁止）
遅く始まる。重く感じる。
ease-out（推奨）
最自然。多くの場面で使う。
Spring（推奨）
cubic-bezier(0.34,1.56,0.64,1)
/* Appleが使うイージング定義 */ :root { --ease-out: cubic-bezier(0.25, 0.46, 0.45, 0.94); --ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1); --ease-decelerate: cubic-bezier(0, 0, 0.2, 1); }

デュレーション（持続時間）

短いアニメーションほどAppleらしい。0.6秒を超えるアニメーションはほぼ存在しない。

カテゴリデュレーション用途
超高速 80–150ms ボタンのActive状態（scale）、マイクロフィードバック
高速 200–300ms ホバー時の色変化、フォーカスリング、ツールチップ出現
標準 300–400ms コンポーネントの展開・折り畳み、モーダルの出現
スロー 500–600ms ページ遷移、スクロールでのコンテンツ出現（上限）
禁止域 600ms以上 ユーザーを待たせる。Appleのプロダクトには存在しない。

体感速度のトリック

同じ300msでも ease-out と linear では速く感じる方が違う。ease-out の方が「始まりが速い」ため体感が短い。Appleが ease-out を選ぶ理由のひとつだ。

アクティブ状態のアニメーション

design.mdに記録されているボタンのアクティブ状態は transform: scale(0.95) だ。これはAppleの最も重要なマイクロインタラクションだ。

正しい実装（押してみて）

scale(0.95) · 150ms · ease-out

誤った実装（押してみて）

opacity変化 → 「暗くなる」だけ

.button-primary { transition: transform 0.15s cubic-bezier(0.25, 0.46, 0.45, 0.94); } .button-primary:active { transform: scale(0.95); } /* transition はactiveプロパティのみに限定する */ /* transition: all はパフォーマンス劣化の原因 */

scale(0.95) が正しい理由：

  • 触覚的フィードバック — 「物理的に押し込む」感覚を視覚で補完する。
  • 全体が縮む — opacity と異なり、形状が変わる。よりリアルな「押した」感。
  • GPU合成レイヤー — transform はGPUで処理されるため、再描画が発生せずパフォーマンスが高い。

スクロールアニメーション

apple.comのスクロール演出は主にフェードイン + 上昇（translateY）の組み合わせだ。

スクロール前（画面外）
opacity: 0 · translateY(20px)
スクロール後（画面内）
opacity: 1 · translateY(0) · 500ms ease-out
/* Intersection Observer でスクロール検出 */ .reveal { opacity: 0; transform: translateY(20px); transition: opacity 0.5s cubic-bezier(0.25, 0.46, 0.45, 0.94), transform 0.5s cubic-bezier(0.25, 0.46, 0.45, 0.94); } .reveal.is-visible { opacity: 1; transform: translateY(0); }

translateY(20px) という値が重要だ。大きすぎると「飛び込んでくる」印象になり主張しすぎる。20〜24pxが「あったかどうかわからない」絶妙な値だ。

Reduced Motion — アクセシビリティ対応

前庭障害や光過敏症を持つユーザーにとって、アニメーションは身体的に辛い体験になることがある。Appleはこれを深刻に扱う。

OSの「視差効果を減らす」設定がオンのとき、prefers-reduced-motion: reduce メディアクエリが発動する。

/* Reduced Motion 対応 — 必須実装 */ @media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; transition-duration: 0.01ms !important; } /* スクロールrevealは即座に表示 */ .reveal { opacity: 1; transform: none; } }

必須チェックリスト

あらゆるアニメーション実装の最後に prefers-reduced-motion ブロックを追加する。これを省略することは、一部のユーザーを排除することだ。

演習

モーション実装と監査

以下のアニメーションをCSSで実装し、Reduced Motion対応まで完成させてください。

  1. button-primary の :active に scale(0.95) を 150ms ease-out で実装する。スマホの実機でタップして確認する。
  2. 3つの要素にスクロール出現アニメーション（opacity + translateY(20px)、500ms ease-out）を Intersection Observer で実装する。
  3. @media (prefers-reduced-motion: reduce) ブロックを追加し、全アニメーションをオフにする。MacのシステムPref→アクセシビリティで「視差効果を減らす」をオンにして動作確認する。
  4. ease-out と linear を同じデュレーションで実装し、並べて比較する動画を録画してコミュニティに投稿する。
  5. 「このモジュールを学ぶ前に使っていたイージング値」と「今後使う値」を1行ずつ書く。
