Module 02 · カラー

カラーシステム構築

Appleが使うカラーは16色以下だ。しかしその選択は偶然ではなく、精密な役割設計の産物。すべての色に「なぜここにあるのか」という理由がある。

このモジュールで学ぶこと

カラーシステムの設計でよくある失敗は、「好きな色を集める」ことだ。ブランドカラーを複数決め、アクセントを数色用意し、状態ごとに別の色を当てていくと、気づけば20色を超えるカラーパレットが出来上がる。

Appleはその正反対だ。インタラクティブな要素に使う色は1色だけ。サーフェスは白・オフホワイト・ニアブラックの3系統のみ。テキストは2色。ボーダーは2色。合計しても15色程度に収まる。そしてこの少なさこそが、一貫性と高級感の源泉だ。

このモジュールの目標

Apple的カラーシステムの全トークンを暗記するのではなく、「なぜその色がそこにあるのか」という設計ロジックを体得すること。

トークン思考：HEXより先にある理解

#0066cc という値を覚えることは簡単だ。しかし本当に必要なのは、「この色はインタラクティブな要素専用であり、それ以外の場所に使ってはいけない」という役割の理解だ。

デザイントークンとは、色に「名前と役割」を与えるシステムだ。{colors.primary} というトークン名は、「ライトサーフェス上のプライマリアクション色」という意味を内包している。HEXが変わっても（例えばブランドリニューアルで青が変わっても）、トークンの役割は変わらない。

色を管理するのではなく、
色の役割を管理する。

デザイントークンの本質

以下のモジュールでは、各カラートークンを「役割」から理解していく。HEXは参考情報として提示するが、覚えるべきはトークン名と使用条件だ。

ブランド・アクセントカラー

Apple的カラーシステムの最大の特徴は、インタラクティブカラーが1色しか存在しないことだ。すべてのリンク、ボタン、フォーカスリングは同じ青系統の色を使う。

なぜ1色だけなのか

複数のアクセントカラーを使うと、ユーザーは「赤いボタンと青いリンクの違いは何か？」を学習しなければならない。単一のアクセントカラーなら、「青いものはクリックできる」という1つのルールだけで完結する。認知負荷が下がり、操作の確信が上がる。

重要な使い分け

ピルCTAはダーク背景でも #0066cc のまま使える（白テキストが十分なコントラストを持つため）。切り替えが必要なのはテキストリンクとインラインコールアウトのみ。

サーフェスカラー

Appleのページはほぼ3系統の背景色で構成される。白・オフホワイト（パーチメント）・ニアブラックだ。それぞれが複数の微差バリアントを持ち、隣接するタイル同士の「かすかな分離」を作り出す。

なぜ3段階のダークが必要か

隣接する2枚のダークタイルを同じ色にすると、境界が消えて1枚の長いダーク面に見えてしまう。2〜3px の輝度差を作ることで、「別々のセクション」という認識を維持できる。

テキストカラー

Appleはテキストカラーも最小限に抑える。ライトサーフェス上は実質1色のニアブラック、ダーク面は白と白の薄いバリアントだけだ。

Text Colors
Near-Black Ink {colors.ink}
#1d1d1f
すべての見出し・本文・ダークユーティリティボタンの塗り。純黒でなくニアブラックを選ぶのは、写真的な質感を保つため。
Body On Dark {colors.body-on-dark}
#ffffff
ダークタイルとグローバルナビ上のすべてのテキスト。
Body Muted {colors.body-muted}
#cccccc
ダークタイル上のセカンダリコピー。純白では主要コピーと区別できないため、グレーに落とす。
Ink Muted 80 {colors.ink-muted-80}
#333333
Pearl Button (#fafafc) の上のテキスト。白に近い背景でも読めるよう、インクを少しだけ和らげる。
Ink Muted 48 {colors.ink-muted-48}
#7a7a7a
無効化ボタンのテキスト・法的免責事項のファインプリント。存在するが主張しない色。

なぜ純黒 (#000000) ではないのか

純黒のテキストは「印刷物」の印象を与える。Appleが目指すのは「写真的な」洗練さだ。#1d1d1f はモニター上でほぼ黒に見えるが、純黒より柔らかく、白背景との対比がわずかに和らぐ。細かい差だが、ページ全体のトーンを決定づける。

ヘアラインとボーダー

Appleがボーダーを使う場面は限られている。ユーティリティカード・コンフィギュレーターチップ・セカンダリボタンの「リング」だ。そのどれもが存在を主張しない薄さで実装されている。

グラデーション禁止の理由

apple.comには装飾的なCSSグラデーションが存在しない。これはデザインドキュメントに明文化されているルールだ。グラデーションのデザイントークンも定義されていない。

なぜか。Appleにとって「大気感」と「奥行き」は写真が担うものだ。iPhone 17 Proのカメラプレートの輝き、Apple Watchバンドの反射、AirPodsの曲面 — これらはすべて、プロダクト写真の中に内在している。CSSグラデーションで後から大気感を足す必要がない。

左のグラデーション例は「トレンドっぽく」見えるかもしれないが、プロダクトのフォルムと競合する。右の単色ダークタイルは無声の台座だ。プロダクト写真（このデモでは省略）が乗れば、背景の存在は消える。

実装パターン

カラートークンを CSS カスタムプロパティとして実装することで、将来のブランド変更にも耐えられるシステムを構築できる。

/* カラートークン定義 */
:root {
  /* Brand & Accent */
  --color-primary:          #0066cc;
  --color-primary-focus:    #0071e3;
  --color-primary-on-dark:  #2997ff;

  /* Surface */
  --color-canvas:           #ffffff;
  --color-canvas-parchment: #f5f5f7;
  --color-surface-tile-1:   #272729;
  --color-surface-black:    #000000;

  /* Text */
  --color-ink:              #1d1d1f;
  --color-body-on-dark:     #ffffff;
  --color-ink-muted-48:     #7a7a7a;
}

/* 使用例 */
.btn-primary {
  background: var(--color-primary);
  color: var(--color-body-on-dark);
}
.tile--dark {
  background: var(--color-surface-tile-1);
  color: var(--color-body-on-dark);
}
.tile--dark a {
  color: var(--color-primary-on-dark); /* Sky Link Blue */
}

Figmaでの実装

Figma Variables（またはTokens Studio プラグイン）で同じ名前のトークンを定義すること。コードとデザインが同じトークン名を参照することで、「デザイナーの意図」と「実装の結果」のズレがなくなる。

演習

カラートークンシートを構築する

Figma（またはスプレッドシート）を開き、このモジュールで学んだカラーシステムを自分の手で再現してください。目的は「全色を暗記する」ではなく、「どのグループに何色あり、各色の役割が何か」を説明できるようになることです。

  1. Figmaで新規ファイルを作成し、「Color System」フレームを用意する（1440×900px）。
  2. Brand & Accent / Surface / Text / Hairline の4グループに分けてカラースウォッチを並べる。
  3. 各スウォッチにトークン名（例: {colors.primary}）とHEXと用途の1行説明をつける。
  4. Variables 機能でトークンを定義し、ライト/ダーク面の切り替えモードを設定してみる。
  5. 「このカラーシステムで作れないもの」を1つ挙げ、その理由を言語化してコミュニティに投稿する。
