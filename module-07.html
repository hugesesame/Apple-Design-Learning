Module 07 · レスポンシブ

レスポンシブ設計

Appleのレスポンシブは「縮小」ではない。画面幅に応じてコンテンツの見せ方を再設計している——8つのブレークポイントがその設計言語だ。

「縮小」vs「再設計」

よくある失敗パターンは「デスクトップをそのまま縮小する」アプローチだ。Appleはこれをしない。

例えばiPhone 15 Proのプロダクトページは、デスクトップ版とまったく異なるレイアウト構造を持つ。ヒーロービジュアルのアスペクト比、CTAの配置、フォントサイズの降下量、ナビゲーションの構造——すべてがモバイル専用に再設計されている。

コンテンツを縮小するな。
そのコンテキストに合わせて再編集せよ。

これがAppleのレスポンシブ設計原則だ。

8つのブレークポイント

design.mdに記録されているブレークポイントは8つ。それぞれに意味がある。

重要

最大コンテナ幅の 1068px は design.md が「Responsive / container-max」として定義する数値だ。これ以上広げてはならない。広げると読者の視線移動距離が長くなりすぎ、Apple的な「余白の美学」が崩れる。

グリッドの崩れ方

Appleのグリッドは「コンテンツを隠す」ことなく崩れる。列数が減るにつれて、コンテンツは積み上がっていく。

アクセサリ・ストアグリッドの具体的崩れ方

ストアのアクセサリグリッドは特に複雑な崩れ方をする。

@media (min-width: 1068px) → grid-template-columns: repeat(4, 1fr);
@media (max-width: 1067px) → grid-template-columns: repeat(3, 1fr);
@media (max-width: 735px) → grid-template-columns: repeat(2, 1fr);
@media (max-width: 419px) → grid-template-columns: 1fr;

タッチターゲット

Appleのタッチターゲット最小サイズは44×44pxだ。これはApple Human Interface Guidelinesの数値と一致する。

Global Nav の高さが44pxに設定されているのはタッチターゲットの最小値と一致するからだ。ナビ内のすべてのリンクは44×44pxのタッチ領域を確保している。

実装テクニック

視覚的なサイズが小さくてもタッチターゲットを大きくする方法: padding で当たり判定を広げる。または ::after 擬似要素で透明な44×44pxの領域を追加する。

ナビゲーションの折り畳み

Global Navのリンクは834px以下で折り畳まれてハンバーガーメニューになる。Sub Navはモバイルで異なる構造に変化する。

演習

ブレークポイント監査と再設計

あなたの手元のプロジェクトまたはapple.comの任意のページを対象に、8つのブレークポイントで表示を確認し、モバイル版の再設計を行ってください。

  1. DevToolsのレスポンシブモードで、1440px / 1068px / 833px / 419px の4サイズで同じページをスクリーンショットする。
  2. 各サイズで「崩れている」か「意図通り変化している」かを分類する。崩れとは：テキストが重なる、要素がはみ出す、余白がゼロになる、タッチターゲットが44px未満になる。
  3. 最も問題のある1つのコンポーネントを選び、4つのブレークポイント全てで正しく機能するようCSSを修正する。
  4. 修正後のコードをFigmaのResponsive Frameで再現し、各ブレークポイントのバリアントを作成する。
  5. 「デスクトップ→モバイルで何が変わったか」を3点でまとめてコミュニティに投稿する。
